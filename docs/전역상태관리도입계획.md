# 전역 상태 관리 도입 계획

## 1. 현재 상태 분석

### 1.1 주요 상태 관리 현황
- 인증/사용자 정보: `useAuth` 훅과 로컬 스토리지 사용
- 카테고리 관리: `useCategories` 훅 사용
- 목표 관리: 컴포넌트 단위의 로컬 상태
- 게시판: 페이지 단위의 데이터 fetching
- AI 채팅: 컴포넌트 내부 상태 관리

### 1.2 현재 문제점
- 컴포넌트 간 상태 공유를 위한 prop drilling
- 데이터 캐싱 및 재사용의 비효율
- 서버 상태와 클라이언트 상태의 구분이 모호
- 상태 업데이트 로직의 중복

## 2. 도입 전략

### 2.1 단계별 접근
1. **1단계: 서버 상태 관리 개선 (1-2주)**
   - React Query 도입
   - API 응답 캐싱
   - 데이터 prefetching
   - 낙관적 업데이트 구현

2. **2단계: 클라이언트 상태 관리 도입 (1-2주)**
   - Zustand 도입
   - 전역 UI 상태 관리
   - 사용자 설정 관리
   - 테마/다크모드 지원

3. **3단계: 리팩토링 및 마이그레이션 (2-3주)**
   - 컴포넌트별 점진적 마이그레이션
   - 테스트 및 버그 수정
   - 문서화 및 가이드라인 작성

### 2.2 우선순위별 마이그레이션 계획

#### 높은 우선순위
1. **인증/사용자 정보**
   - 전역 상태로 관리
   - 새로고침 시 상태 유지
   - 권한 관리 통합

2. **카테고리 및 목표 데이터**
   - 서버 상태 캐싱
   - 실시간 업데이트
   - 목록/상세 데이터 정규화

#### 중간 우선순위
3. **게시판 데이터**
   - 페이지네이션 상태 관리
   - 필터링/정렬 상태
   - 임시 저장 기능

4. **AI 채팅**
   - 채팅 히스토리 관리
   - 모델 설정 상태
   - 스트리밍 상태 관리

#### 낮은 우선순위
5. **UI 상태**
   - 테마 설정
   - 레이아웃 설정
   - 사용자 기본 설정

## 3. 기술 스택 선정

### 3.1 서버 상태 관리
- **React Query**
  - 이유: 
    - 서버 상태 특화
    - 강력한 캐싱 시스템
    - Next.js와의 높은 호환성
    - 낙관적 업데이트 지원

### 3.2 클라이언트 상태 관리
- **Zustand**
  - 이유:
    - 가벼운 번들 사이즈
    - 간단한 학습 곡선
    - TypeScript 지원
    - 미들웨어 시스템

## 4. 구현 가이드라인

### 4.1 상태 구분 원칙
- 서버 상태: API 응답 데이터
- 클라이언트 상태: UI 상태, 사용자 설정
- 로컬 상태: 폼 입력값, 임시 데이터

### 4.2 코드 구조
```
/src
  /stores
    /server  // React Query hooks
      useCategories.ts
      useGoals.ts
      useBoards.ts
    /client  // Zustand stores
      useAuthStore.ts
      useUIStore.ts
      useSettingsStore.ts
```
